## Context

当前好友中心（`friend-panel.tsx`）使用自定义 CSS 类名体系（`.friend-panel`, `.friend-sidebar`, `.friend-card` 等），与使用 shadcn/ui + Tailwind 的会话侧边栏（`conversation-sidebar.tsx`）在视觉风格上不一致。

桌面端应用同时存在两个主要工作区：聊天界面和好友中心。用户在两者之间切换时，UI 风格的割裂感影响使用体验。

## Goals / Non-Goals

**Goals:**
- 好友中心与聊天侧边栏视觉风格统一
- 复用 shadcn/ui 组件库，减少自定义 CSS
- 简化布局：两栏结构替代原有三栏

**Non-Goals:**
- 不改变任何业务逻辑和功能
- 不新增 API 接口
- 不修改移动端 UI（仅限桌面端）
- 不引入新的依赖库

## Decisions

### 1. 迁移到 shadcn/ui + Tailwind

**决策**：好友中心全面采用 shadcn/ui 组件和 Tailwind CSS 类名

**理由**：
- ConversationSidebar 已采用此技术栈，统一后降低认知负担
- shadcn/ui 组件提供一致的交互体验（hover、focus、disabled 状态）
- Tailwind 原子化 CSS 便于统一间距、颜色、圆角等设计 token

**替代方案**：
- 保留部分自定义 CSS，仅统一颜色变量 → 效果有限，仍有组件层面不一致
- 完全重写好友中心，同时重构 App.tsx 中的组合逻辑 → 改动范围过大

### 2. 两栏布局替代三栏

**决策**：左侧 280px 固定宽度联系人列表 + 右侧 flex-1 详情操作区

**理由**：
- 与 ConversationSidebar 宽度一致（w-[280px] min-w-[280px]）
- 桌面端宽度充足，两栏布局更舒展
- 右侧区域可展示联系人详情、关系总览、操作按钮

**替代方案**：
- 保留三栏（侧边栏导航 + 列表 + 详情）→ 导航功能被 Tab 替代后可压缩
- 单栏 + Modal 详情 → 破坏桌面端信息密度优势

### 3. Tab 导航替代左侧边栏分类

**决策**：顶部 Tab 切换「好友 / 待处理 / 黑名单」三个分类

**理由**：
- 三个分类是互斥的，不需并行展示，Tab 是更自然的交互模式
- 释放左侧空间给联系人列表
- 与现代应用常见模式一致

### 4. 保留功能清单

| 功能 | 处理方式 |
|------|----------|
| 搜索联系人 | 顶部搜索框，与 ConversationSidebar 样式一致 |
| 联系人列表 | 复用 `conversation-card` 样式 |
| 联系人详情 | 右侧区域展示 |
| 发起聊天 | 「发消息」按钮 |
| 同意/拒绝请求 | 「同意」「拒绝」按钮 |
| 拉黑/解除拉黑 | 「拉黑」「解除黑名单」按钮 |
| 关系总览 | 右侧区域展示统计数字 |

### 5. 移除功能清单

| 功能 | 移除原因 |
|------|----------|
| 二维码加好友 | 桌面端不实用，且涉及复杂 QR 库依赖 |
| 复制加好友码 | 同上，桌面端用户不会使用此方式 |
| 批量处理请求 | 桌面端场景下使用频率低 |

## Risks / Trade-offs

| 风险 | 影响 |  Mitigation |
|------|------|-------------|
| 迁移过程中引入布局 bug | 好友中心功能异常 | 在设计阶段明确各区域宽高、间距规范 |
| 移除二维码功能后有用户反馈 | 部分用户依赖此功能 | 记录在案，后续可按需加回（改动成本低） |
