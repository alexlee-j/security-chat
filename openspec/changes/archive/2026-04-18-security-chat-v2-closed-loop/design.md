## Context

当前仓库已经完成了 1.x 向“安全聊天基础可用态”的跨越：直聊多设备、Rust Signal 主链路、群聊 Sender Key 基础路径、通知设置和回归矩阵都已有落地。但从产品视角看，系统仍缺少几个关键闭环：

- 用户注册后缺少完整的账户恢复链路，密码策略与忘记密码尚未收口。
- 群聊只有“能建群、能发收”的基础路径，缺少群资料管理、成员治理、退出/移除/重入等完整产品行为。
- 桌面端本地消息收发和重连恢复仍有 TODO / 占位逻辑，真实使用时会触发同步、失败恢复和摘要不一致问题。
- 当前发布说明偏向阶段性验收，不足以支撑“2.0 正式版”。

这个变更是跨 backend / desktop / Rust / docs 的交叉方案，且涉及鉴权、安全恢复、协议生命周期和版本门槛，因此需要先把技术决策写清楚，再进入实现。同时需要控制阶段范围：桌面端总体设计文档覆盖了更广的产品表层能力，但本变更只处理聊天产品 2.0 核心闭环，不把“桌面产品面全兑现”与“音视频通话”混入同一阶段。

## Goals / Non-Goals

**Goals:**
- 完成账户安全闭环：注册密码规则、找回密码、密码重置后的设备与会话处理、异常登录恢复。
- 完成群聊产品闭环：群资料、成员管理、退出/踢人/重入规则与 Rust Sender Key 生命周期对齐。
- 完成会话同步闭环：本地消息状态机、断线补拉、失败重试、消息回显与会话摘要一致。
- 完成 2.0 发布闭环：明确 release gate、运维说明和文档边界。
- 明确角色分工与分支治理，保证后续桌面产品面闭环以独立 change 和独立分支推进。

**Non-Goals:**
- 不在本变更中引入音视频通话、屏幕共享或其他 V3 范围能力。
- 不在本变更中完成桌面端除聊天核心外的全部产品表层设计兑现；这部分作为后续独立 change 处理。
- 不重写现有直聊或群聊密码学方案；本次只在现有 Rust-first 路径上补齐生命周期与产品面。
- 不引入新的桌面端技术栈或替换 Tauri / React / NestJS 主架构。

## Decisions

### 1. 2.0 范围定义为“四条闭环”，不继续做零散功能修补

2.0 的目标不是再做几个离散 feature，而是让产品在真实使用中不再出现“入口有了、退出路径没有”或“加密能跑、状态不闭环”的断层。因此范围按四条闭环定义：

1. 账户恢复闭环
2. 群治理闭环
3. 同步可靠性闭环
4. 发布运维闭环

这样拆分后，需求、代码与回归门槛可以对应到同一个版本目标。

备选方案：
- 继续按模块逐个修：容易回到“局部完成、整体不闭环”的状态。
- 直接推进 V3 音视频：当前基础产品面尚未闭环，会放大复杂度和回归成本。

### 2. 账户恢复沿用现有认证体系扩展，不引入独立身份系统

后端已经有 `auth`、`mail`、`user`、设备绑定 JWT 和多端语义，因此 2.0 的账户恢复应建立在现有体系上：

- 注册时执行统一密码策略。
- 忘记密码使用邮件验证码或同级恢复令牌。
- 密码重置后显式处理刷新 token、设备会话与重新登录提示。

备选方案：
- 单独引入外部认证系统：实现成本高，且会与现有设备绑定语义冲突。
- 只补前端密码校验：不能解决真实恢复与安全事件一致性问题。

### 3. 群治理必须与 Sender Key 生命周期统一建模

群聊产品行为不能独立于协议层。成员加入、移除、退出、重入这些动作不只是 UI 事件，而是密码学状态切换点。因此群治理设计采用“产品事件即协议事件”的模型：

- 成员变更后，后端和 Rust 层必须有明确的 Sender Key 轮换/再分发规则。
- 前端展示与服务端状态必须围绕同一生命周期事件，例如“被移除后只能看历史不能看未来消息”。

备选方案：
- 先做 UI 群管理，协议后补：会在真实场景中制造安全语义错位。
- 只做协议轮换，不补产品提示：用户无法理解群状态变化。

### 4. 同步可靠性采用“逻辑消息状态机 + 显式 outbox/retry”模式

桌面端当前有本地数据库，但上传/补拉仍有 TODO。2.0 需要把消息生命周期正式化：

- 本地消息至少区分 `draft / queued / sending / sent / failed / replayed`
- 网络恢复后按 conversation / cursor 补拉历史
- 失败重试必须显式且幂等，不能靠静默回退
- 会话摘要使用统一来源：metadata-first 或可解密缓存摘要，禁止多套策略并存

备选方案：
- 继续即时直发、失败即报错：可演示，但不满足正式版本体验。
- 后端强行兜底所有重试：无法覆盖桌面端本地状态和 UX 反馈。

### 5. 2.0 发布门槛升级为“版本运行条件”，不再只是阶段性检查表

现有 release readiness 更偏一次性变更交付。2.0 要把这部分升级为持续可执行的版本治理：

- backend / Rust / desktop / 真实账号冒烟必须形成稳定矩阵
- README、运维脚本、回滚说明必须跟随版本更新
- 合并门槛应围绕 2.0 产品闭环，而不是只围绕单项功能 green

备选方案：
- 保持当前文档状态：后续版本会继续依赖口头约定。

### 6. 协作采用“高风险由 GPT 收口，低风险由 MiniMax 承担”的双角色模型

当前变更横跨认证、安全、协议、同步状态机和桌面接入。为了避免高风险逻辑与表层整合同步失控，本设计明确：

- GPT 负责重点难点功能：认证恢复与安全边界、群治理与 Sender Key 生命周期、消息同步可靠性、跨层状态一致性、发布门槛与回滚。
- MiniMax 负责支撑性事项：低风险 UI 接入、状态展示、基础页面整合、说明文档与配套验证材料。

备选方案：
- 不区分角色、按文件随机分配：容易导致高风险逻辑被表层实现带偏。

### 7. 新阶段必须从 main 基线重新分支，不允许在未合并的旧阶段分支上叠新 change

本变更完成后，后续“桌面产品面闭环”会成为独立 change。若下一阶段不是继续实现当前 2.0 change，而是进入新 change，则必须遵守：

1. 先完成当前分支的验证与合并；
2. 将当前实现分支合入 `main`；
3. 从最新 `main` 创建新分支；
4. 在新分支上创建并实现新的 OpenSpec change。

这样做的理由是保持：
- git 分支与 OpenSpec change 一一对应；
- 回滚与发布基线清晰；
- 避免把两个阶段的风险混在同一长寿命分支中。

## Risks / Trade-offs

- [账户恢复触发设备会话处理复杂] → 通过“密码重置后显式失效 refresh token、要求重新登录、保留可配置设备保留策略”降低风险。
- [群治理与 Sender Key 耦合提高实现成本] → 先定义成员生命周期事件，再让 backend / Rust / frontend 依同一事件实现，避免三端各自解释。
- [本地同步状态机会带来迁移成本] → 通过渐进式字段扩展和幂等重试接口，避免一次性重写本地库。
- [2.0 范围过大] → 任务拆解以闭环为单位推进，每条闭环都必须可独立验证，避免“大而全但无法落地”。
- [桌面产品面需求继续膨胀] → 将其明确为后续独立 change，不在当前阶段扩容。

## Migration Plan

1. 新建 2.0 集成变更分支，在该分支内完成全部实现与回归。
2. 先完成 spec-level 行为收口，再按账户恢复、群治理、同步可靠性、发布运维顺序实施。
3. 对需要兼容迁移的路径保留短暂窗口，但必须有明确去除条件，不允许长期双轨。
4. 文档、脚本、README 与回归矩阵跟代码同步修改。
5. 只有当 backend / Rust / desktop / 真实账号冒烟全部通过时才允许进入 2.0 合并评审。
6. 若进入下一阶段桌面产品面闭环，则必须在当前 change 合并到 `main` 后，从 `main` 重新拉分支。

回滚策略：
- 账户恢复、群治理、同步可靠性三条主闭环分别按独立里程碑提交，出问题时按里程碑回滚。
- 不允许通过手工删文件或临时关闭校验来伪造 2.0 完成态。

## Open Questions

- 密码重置是否要求“一次性踢出所有设备”，还是允许保留当前发起设备？
- 群治理的最小权限模型是“群主唯一管理”，还是需要引入管理员角色？
- 会话摘要在端到端加密前提下，2.0 是否接受 metadata-first 占位文案，还是必须实现本地可解密摘要缓存？
- 桌面端同步状态机是否需要同步暴露给移动端，还是先在桌面端完成验证后再推广？
- 后续独立的桌面产品面闭环 change，是否需要把 `profile/settings/about` 与搜索、上下文菜单、输入增强作为一个能力簇一起推进？
