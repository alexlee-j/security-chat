## Context

桌面端当前存在三类交互分裂：登录页的记住密码/自动登录状态由本地表单和持久化存储分别维护，好友中心维护了自己的汉堡菜单和导航抽屉，右键菜单则把本地态动作与支持态动作混在一起展示。这个 change 目标不是增加新业务能力，而是把当前桌面产品面收束成单一交互模型，并让好友中心的视觉和信息结构与主页面保持一致。

## Goals / Non-Goals

**Goals:**
- 让登录页的“记住密码”和“自动登录”成为一组一致的状态，而不是两个互不关联的勾选项。
- 让会话列表和好友中心共享同一个全局导航入口与抽屉行为。
- 让好友中心重构成与当前桌面整体设计一致的页面结构，避免像独立子产品。
- 让会话列表右键菜单只展示当前可用且语义清晰的动作。

**Non-Goals:**
- 不改变后端认证协议、Signal 协议或消息发送链路。
- 不新增朋友关系、黑名单、会话删除之外的服务端能力。
- 不在本次 change 中引入新的设计系统或重做整体主题系统。

## Decisions

### 1) 登录偏好由 `useChatClient` 作为单一状态源，登录表单做受控展示
当前问题的根源是 `LoginForm` 维护了自己的 `react-hook-form` 状态，而 `useChatClient` 也维护了记住密码与自动登录状态。设计上应该把偏好状态收敛到 `useChatClient`，登录表单只负责展示和提交。

选择原因：
- 避免“提交状态正确、界面状态错误”的双写问题。
- 自动登录恢复时可以直接回填表单并保持 UI 与持久化一致。

备选方案：
- 让 `LoginForm` 自己读取 secure storage。这个方案会把持久化逻辑下沉到表单层，导致职责更乱，也更难复用。

### 2) 自动登录与记住密码建立硬约束
当用户勾选自动登录时，系统必须同步勾选记住密码；当用户取消记住密码时，自动登录必须被取消。这个约束应在状态层和表单层同时成立。

选择原因：
- 这符合用户心智模型，自动登录本质上依赖持久密码状态。
- 避免出现“自动登录打开但密码未保存”的无效配置。

备选方案：
- 允许自动登录独立于记住密码存在。这个方案会制造无效配置和额外错误处理分支。

### 3) 好友中心使用与聊天工作区相同的全局 shell 和导航抽屉
好友中心不再维护自己的独立导航抽屉语义。菜单按钮只是同一个全局导航入口的不同放置点，抽屉状态、内容和关闭行为统一由 App 级管理。

选择原因：
- 降低认知负担，用户在任意工作区都看到同一套应用级导航。
- 避免两个地方各自维护抽屉状态、主题入口和退出登录行为。

备选方案：
- 保留好友中心自己的抽屉实现，只做样式统一。这个方案会继续保留两套状态和代码路径。

### 4) 好友中心采用“两栏内容 + 统一头部”的页面结构
好友中心应保留当前页面总体设计里的信息层级：顶部应用标题/入口，中间左侧列表、右侧详情/操作卡片。布局以当前桌面主页面的卡片、分割线、圆角、间距和颜色变量为准，而不是引入独立视觉语言。

选择原因：
- 这是桌面端当前整体设计最稳定的一部分，用户切换工作区时不会感到跳脱。
- 左右两栏布局更适合展示好友搜索、请求列表、黑名单和详情动作。

备选方案：
- 做成单栏抽屉式好友中心。这个方案更轻，但会降低信息密度和操作效率。

### 5) 会话列表右键菜单采用能力矩阵驱动
菜单展示与否由会话类型、当前状态、支持能力共同决定。Local-only 动作可以保留，但必须显式命名；不可用动作必须隐藏或禁用，并给出明确理由。

选择原因：
- 避免“看起来可点、实际上没效果”的假动作。
- 让上下文菜单与 `ChatMoreMenu`、删除确认对话框保持一致的语义边界。

备选方案：
- 继续展示所有动作，但由点击后兜底提示。这会让 UI 继续承载无效入口。

## Risks / Trade-offs

- [Risk] 登录表单改成受控状态后更容易出现重渲染和 reset 时机问题。→ Mitigation: 把表单 reset 限定在持久化状态真正变化时，避免每次输入都触发。
- [Risk] 好友中心共享全局 shell 后，App 级状态会更集中。→ Mitigation: 只提升导航抽屉和 workspace 切换状态，不把好友面业务数据上提。
- [Risk] 右键菜单能力矩阵如果设计过细，会增加判断复杂度。→ Mitigation: 先按“会话类型 / 设备支持 / 本地态 / 服务端态”四类做最小分层。
- [Risk] 改动会影响现有桌面端视觉一致性。→ Mitigation: 复用当前页面的颜色变量、卡片边界和 shadcn/Radix 组件，不引入新视觉语言。

## Migration Plan

1. 先改登录偏好状态链路，让表单、持久化存储和自动登录恢复在同一套状态模型下工作。
2. 提取或复用一个 App 级全局导航抽屉触发器，替换好友中心内部独立菜单逻辑。
3. 重构好友中心页面结构，保留现有功能但统一成当前桌面设计语言。
4. 收口会话右键菜单的动作可用性判断，并补齐空状态/不可用状态文案。
5. 使用桌面构建和真实交互冒烟验证回归。

## Open Questions

- 自动登录关闭时是否要保留已保存密码，还是同时清空密码缓存？
- 好友中心右侧详情区是否需要加入统一的“更多”菜单，还是保持当前直接操作按钮布局？
- 会话右键菜单中的“复制ID”是否需要在没有剪贴板权限时显示明确不可用提示，还是仅静默失败？
