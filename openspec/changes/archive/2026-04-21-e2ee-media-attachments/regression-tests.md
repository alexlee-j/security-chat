# E2EE 媒体附件回归测试文档

## 测试时间
2026-04-21

## 测试环境
- 前端桌面应用: `pnpm run tauri:dev`
- 后端: `pnpm start:backend:dev`
- Rust 加密库: `cargo test --manifest-path apps/desktop/src-tauri/Cargo.toml`
- 浏览器: Chromium (Playwright)

## 快速验证命令

### 1. Rust 加密单元测试
```bash
cargo test media --manifest-path apps/desktop/src-tauri/Cargo.toml
```
验证：加密/解密圆转、错误拒绝（错误密钥、篡改密文、错误摘要）

### 2. 后端媒体测试
```bash
pnpm -C apps/backend test -- media.copy.spec.ts burn.media.spec.ts
```
验证：加密上传、绑定、下载、复制、焚毁清理

### 3. 桌面应用构建
```bash
pnpm -C apps/desktop build
```
验证：TypeScript 编译、Vite 构建、桌面前端产物

### 4. 后端构建
```bash
pnpm -C apps/backend build
```
验证：NestJS 编译和媒体 DTO/entity/service 类型一致。

### 5. 媒体附件人工烟雾测试

目前仓库没有可直接运行的 `e2e-media-test.cjs` 自动化脚本。桌面端 image/audio/file 媒体冒烟需要在后端和桌面应用启动后手动执行，或后续补 Playwright/Tauri 自动化脚本。

## 详细测试用例

### TC-1: 图片加密上传

**前置条件**：
- 用户已登录
- 已选择或创建会话

**步骤**：
1. 点击附件按钮，选择一张图片
2. 发送图片消息
3. 拦截网络请求，确认 `POST /media/upload` 上传的 body 是密文（非明文图片数据）

**预期结果**：
- ✅ 上传请求的 Content-Type 为 `application/octet-stream`
- ✅ 上传内容无法直接识别为图片（密文无明文 magic bytes）
- ✅ 消息气泡正常显示

**回归检查**：
- ✅ 上传取消功能仍然工作

---

### TC-2: 图片消息接收和预览

**前置条件**：
- 发送方已发送加密图片消息
- 接收方已加入同一会话

**步骤**：
1. 等待图片消息出现
2. 点击图片进行预览

**预期结果**：
- ✅ 消息气泡显示缩略图或加载占位
- ✅ 点击后显示解密后的明文图片
- ✅ 图片预览可关闭

**回归检查**：
- ✅ 阅后即焚图片在触发后正确焚毁
- ✅ 转发加密图片保持加密语义

---

### TC-3: 音频加密上传和播放

**前置条件**：
- 用户已登录并选择会话

**步骤**：
1. 点击附件按钮，选择音频文件
2. 发送音频消息
3. 等待接收方收到
4. 点击播放按钮

**预期结果**：
- ✅ 上传的音频是密文
- ✅ 播放时自动解密
- ✅ 播放控制正常工作

**回归检查**：
- ✅ 语音消息时长显示正确

---

### TC-4: 文件加密上传和下载

**前置条件**：
- 用户已登录并选择会话

**步骤**：
1. 点击附件按钮，选择任意文件
2. 发送文件消息
3. 点击下载按钮

**预期结果**：
- ✅ 上传的文件是密文
- ✅ 下载时解密得到正确文件名
- ✅ 文件可正常打开

**回归检查**：
- ✅ 文件名和 MIME 类型正确保留

---

### TC-5: 解密失败错误状态

**前置条件**：
- 存在加密图片消息

**步骤**：
1. 手动篡改 localStorage 中的媒体密钥（模拟密钥不匹配）
2. 尝试加载图片

**预期结果**：
- ✅ 显示"解密失败，请检查密钥"错误状态
- ✅ 不显示通用的网络错误图标
- ✅ 错误图标为锁定状态

---

### TC-6: 缺少媒体元数据错误状态

**前置条件**：
- 存在媒体 assetId 但 payload.media 缺失

**步骤**：
1. 构造异常消息（assetId 存在但 media 元数据为空）
2. 尝试加载媒体

**预期结果**：
- ✅ 显示"媒体元数据缺失"错误状态

---

### TC-7: 旧版媒体回退

**前置条件**：
- 数据库中存在旧版明文媒体资产

**步骤**：
1. 查找并打开包含旧版 mediaUrl 的历史消息
2. 确认图片/音频/文件可正常加载

**预期结果**：
- ✅ 旧版消息仍可正常查看
- ✅ 不影响新版加密媒体功能

**回归检查**：
- ✅ 旧版媒体下载不经过解密流程

---

### TC-8: 焚毁清理

**前置条件**：
- 发送带阅后即焚的加密图片

**步骤**：
1. 发送阅后即焚图片消息
2. 接收方点击查看（触发焚毁倒计时）
3. 等待倒计时结束或点击焚毁按钮

**预期结果**：
- ✅ 后端正确删除密文媒体资产
- ✅ 前端显示焚毁动画后移除消息

## 已知限制

### KL-1: 旧版明文媒体
- **描述**：数据库中现有的明文媒体资产将继续以明文形式存储和传输
- **影响**：这些消息不享受 E2EE 保护
- **建议**：参见 `legacy-media-compatibility.md` 中的迁移选项

### KL-2: 移动端兼容性
- **描述**：当前 E2EE 媒体实现仅覆盖桌面端
- **影响**：如果用户仅在移动端查看历史加密媒体，将无法解密
- **建议**：后续实现移动端加密媒体支持

### KL-3: 转发语义
- **描述**：转发的加密媒体复用相同的密文和密钥
- **影响**：原始接收者和新接收者共享相同解密材料
- **建议**：未来可考虑转发时重新加密以增强隐私

### KL-4: 大文件流式处理
- **描述**：当前实现不支持流式加密/解密
- **影响**：超大文件可能造成内存压力
- **建议**：未来实现流式处理

### KL-5: 媒体预览对象 URL 生命周期
- **描述**：解密后的 blob URL 需要及时 revoke
- **影响**：如果未正确清理，可能导致内存泄漏
- **建议**：已实现 cleanup 逻辑，监控内存使用
