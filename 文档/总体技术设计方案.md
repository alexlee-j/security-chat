## 1. 设计目标

- 单工程、跨平台一致体验（iOS/Android/macOS/Windows）
- 默认安全：传输加密 + 端到端加密 + 本地加密存储
- 多设备实时同步，离线可补偿
- 阅后即焚严格可审计、可验证
- 支持后续横向扩展与高并发

## 2. 系统总体架构

### 2.1 客户端层

- `apps/mobile`：React Native（iOS/Android）
- `apps/desktop`：Tauri + React（macOS/Windows）
- 共享能力：
  - `packages/ui`（组件）
  - `packages/shared-types`（协议与类型）
  - `packages/state`（状态管理）
  - `packages/crypto`（加解密与密钥流程）
  - `packages/websocket`（长连接封装）

### 2.2 服务端层

- `apps/backend`：NestJS 模块化服务
- 核心模块：
  - Auth（登录、Token、RefreshToken、黑名单）
  - User（用户资料、设备管理）
  - Friend（关系链）
  - Conversation（会话、成员）
  - Message（消息写入与投递）
  - Burn（阅后即焚触发与销毁）
  - Notify（通知）
  - Audit/Security（审计与风控）
- 接入通道：
  - HTTP API（注册登录、资源上传签名、管理接口）
  - WebSocket Gateway（消息与状态实时通道）

### 2.3 数据与中间件层

- PostgreSQL：主业务数据
- Redis：在线状态、会话缓存、发布订阅、限流状态
- 对象存储（MinIO/S3）：图片、语音、文件

## 3. 通信与协议设计

### 3.1 基础协议

- 传输层强制 `TLS 1.3`
- 应用层 JSON 协议，统一 envelope：
  - `trace_id`
  - `event_type`
  - `conversation_id`
  - `device_id`
  - `payload`
  - `server_ts`

### 3.2 消息状态机

- 客户端本地：`sending`
- 服务端落库成功：`sent`
- 目标设备收到：`delivered`
- 目标设备读到：`read`

### 3.3 多端同步机制

- 基于 `conversation_id + message_index` 进行有序增量同步
- WS 断线重连后，按最后确认游标补拉
- 已读状态以用户维度广播到其所有在线设备

## 4. 端到端加密（E2EE）设计

### 4.1 密钥模型

- 每设备持有身份密钥对（Identity Key）
- 服务端保存公开材料（公钥、签名 prekey、one-time prekeys）
- 新会话建立时消耗 one-time prekey

### 4.2 消息加密

- 消息体：`AES-256-GCM` 加密
- 数据库存储：`encrypted_payload + nonce + 元数据`
- 服务端不存私钥、不持会话密钥、不解密正文

### 4.3 客户端安全存储

- iOS：Keychain
- Android：Keystore
- macOS：Keychain
- Windows：Credential Vault

## 5. 阅后即焚设计

### 5.1 触发与计时

- 文本：打开聊天窗口触发
- 图片：打开预览触发
- 语音：开始播放触发

### 5.2 销毁策略

- 支持时长：5s/10s/30s/1m/5m
- 未查看超过 24h 自动销毁
- 销毁后服务端硬删除消息并广播删除事件

### 5.3 一致性与防重

- `burn_events` 记录触发源与触发时间
- 防止重复销毁、重复通知
- 多设备同步执行本地删除

## 6. 数据库设计落地

沿用并实现以下核心表：

- `users`
- `devices`
- `one_time_prekeys`
- `friendships`
- `conversations`
- `conversation_members`
- `messages`
- `burn_events`
- `notifications`

关键约束与建议：

- `friendships`：`unique(user_id, friend_id)`
- `messages`：索引 `(conversation_id, created_at DESC)`
- 消息强序：`message_index`（会话内单调）
- 建议补充字段：`messages.is_deleted`（用于普通消息软删策略）

## 7. 事务与一致性策略

发消息主链路：

1. 校验会话与成员权限
2. 写入 `messages`
3. 提交事务
4. 发布 Redis 事件
5. WS 广播给在线设备

原则：**未落库成功不得推送**

## 8. Redis 设计

- 在线状态：
  - `online:{userId} -> [deviceIds]`
- 连接映射：
  - `ws:connection:{userId}:{deviceId} -> socketId`
- 心跳 TTL：60 秒
- 跨实例分发：Redis Pub/Sub（消息、已读、删除事件）

## 9. 文件与媒体设计

- 文件本体仅存对象存储
- PostgreSQL 保存元数据与访问控制信息
- 阅后即焚媒体支持短期签名 URL，销毁后立即失效

## 10. 安全与风控

- 登录防爆破：IP + 账号维度限流
- WebSocket 鉴权：连接握手校验 Token + 设备 ID
- 防重放：消息请求带 nonce/timestamp，服务端校验窗口
- Token 黑名单：登出、异常登录、风险事件可即时失效
- 审计日志：安全关键事件（登录、密钥更新、销毁触发）

## 11. 性能与扩展方案

- 单节点目标：10 万在线连接
- 网关层无状态化，按连接数横向扩容
- 大数据量后对 `messages` 按月分区
- 后续演进：读写分离、冷热分层、分库分表

## 12. Monorepo 工程规范

- 统一包管理：`pnpm workspace`
- 统一规范：ESLint + Prettier + TypeScript 严格模式
- CI 流水线：
  - lint
  - typecheck
  - unit test
  - e2e test（消息与同步关键路径）

## 13. V1 交付边界

- 单聊
- 普通消息（文本/图片/语音/文件基础能力）
- 阅后即焚（文本/图片/语音）
- 多端实时同步与离线补偿

非 V1（后续）：

- 群聊
- 消息撤回
- 企业版能力

## 14. 验收指标映射

- 文本消息端到端延迟：<= 1 秒
- 同步延迟：<= 3 秒
- 抓包不可见明文
- 本地数据库不可直接明文读取
- 阅后即焚销毁后不可恢复

## 15. 风险与优先级

高风险项（优先攻坚）：

1. 多端一致性（已读、删除、重连补偿）
2. 阅后即焚严格销毁链路
3. 密钥生命周期与设备变更
4. 弱网场景稳定性
5. 高并发 WS 扩展
