**文档版本**: v2.0 (Electron + Signal 官方库)

## 一、设计目标

- **跨平台一致体验**: 单工程构建，支持 macOS/Windows/Linux/iOS/Android
- **默认安全**: 端到端加密 (Signal 官方库) + 传输加密 + 本地加密存储
- **多设备实时同步**: 离线补偿机制
- **阅后即焚**: 严格可审计、可验证
- **可扩展**: 支持后续横向扩展与高并发

## 二、系统架构

### 2.1 客户端层

| 平台 | 技术栈 | Signal 协议库 |
|------|--------|--------------|
| **桌面端** | Electron + React | `@signalapp/libsignal-client` (官方库) |
| **移动端** | React Native | `react-native-libsignal` (待集成) |

**共享模块**:
- `packages/ui` - UI 组件库
- `packages/shared-types` - 协议与类型定义
- `packages/state` - 状态管理
- `packages/websocket` - WebSocket 长连接封装

### 2.2 服务端层

- **框架**: NestJS 11
- **核心模块**:
  - Auth - 认证 (JWT, RefreshToken)
  - User - 用户资料、设备管理
  - Friend - 好友关系
  - Conversation - 会话管理
  - Message - 消息路由
  - Burn - 阅后即焚
  - PreKey - 预密钥分发

### 2.3 数据层

- **PostgreSQL**: 主业务数据
- **Redis**: 在线状态、会话缓存、Pub/Sub
- **对象存储**: 图片、语音、文件 (MinIO/S3)

## 四、Signal 协议实现

### 4.1 技术方案

| 组件 | 实现 |
|------|------|
| **协议规范** | X3DH + Double Ratchet (Signal 官方) |
| **桌面端库** | `@signalapp/libsignal-client@0.90` |
| **加密算法** | AES-256-GCM |
| **密钥交换** | ECDH P-256 |
| **签名算法** | ECDSA P-256 |
| **密钥派生** | HKDF (HMAC-SHA256) |

### 4.2 X3DH 密钥交换流程

```
发送方 (Alice)                          接收方 (Bob)
    │                                      │
    │  1. 从服务器获取 Bob 的预密钥包          │
    │◄─────────────────────────────────────│
    │                                      │
    │  2. 生成临时密钥对 (Ephemeral)         │
    │                                      │
    │  3. 计算 4 次 DH 共享密钥:              │
    │     DH1 = DH(身份私钥，签名预密钥公钥)   │
    │     DH2 = DH(临时私钥，身份公钥)        │
    │     DH3 = DH(临时私钥，签名预密钥公钥)  │
    │     DH4 = DH(临时私钥，一次性预密钥公钥)│
    │                                      │
    │  4. SK = KDF(DH1 || DH2 || DH3 || DH4)│
    │                                      │
    │  5. 发送加密消息 (包含临时公钥)         │
    │──────────────────────────────────────►│
    │                                      │
    │                          6. 计算共享密钥:
    │                             DH1 = DH(签名预密钥私钥，发送方身份公钥)
    │                             DH2 = DH(身份私钥，发送方临时公钥)
    │                             DH3 = DH(签名预密钥私钥，发送方临时公钥)
    │                             DH4 = DH(一次性预密钥私钥，发送方临时公钥)
    │                                      │
    │                          7. SK = KDF(DH1 || DH2 || DH3 || DH4)
    │                                      │
    │                          8. 解密消息
```

### 4.3 Double Ratchet 消息加密

```
发送消息:
1. 检查会话状态
2. 如需执行 DH 棘轮
3. 推进发送链
4. 派生消息密钥
5. AES-GCM 加密
6. 发送密文

接收消息:
1. 检查 DH 棘轮
2. 推进接收链
3. 派生消息密钥
4. AES-GCM 解密
5. 更新会话
```

### 4.4 密钥层次结构

```
Identity Key Pair (长期密钥)
├── ECDH Key Pair (密钥交换)
└── ECDSA Key Pair (签名)
    │
    └── Signed PreKey (签名预密钥)
        │
        └── One-Time PreKeys (一次性预密钥，100 个)
```

## 五、通信协议

### 5.1 传输层

- **TLS 1.3**: 强制加密传输
- **WebSocket**: 实时消息推送
- **HTTP API**: 注册登录、资源上传

### 5.2 应用层协议

统一 Envelope 格式:
```json
{
  "trace_id": "uuid",
  "event_type": "message.new",
  "conversation_id": "uuid",
  "device_id": "uuid",
  "payload": "...",
  "server_ts": 1234567890
}
```

### 5.3 消息状态机

```
客户端 → sending → 服务端落库 → sent → 对方接收 → delivered → 对方读取 → read
```

## 六、数据库设计

### 6.1 核心表

| 表名 | 用途 |
|------|------|
| `users` | 用户信息 |
| `devices` | 设备信息 |
| `identity_keys` | 身份公钥 |
| `signed_prekeys` | 签名预密钥 |
| `one_time_prekeys` | 一次性预密钥 |
| `conversations` | 会话 |
| `conversation_members` | 会话成员 |
| `messages` | 消息 |
| `burn_events` | 阅后即焚事件 |

### 6.2 关键约束

- `messages(conversation_id, message_index)` - 唯一索引
- `messages(conversation_id, created_at DESC)` - 查询索引
- `one_time_prekeys(user_id, key_id)` - 唯一索引

## 七、安全设计

### 7.1 端到端加密

- 消息体使用 AES-256-GCM 加密
- 服务端不存储私钥、不解密消息
- 数据库存储：`encrypted_payload + nonce + metadata`

### 7.2 客户端存储

| 平台 | 存储方式 |
|------|----------|
| Electron 主进程 | 加密后存储 (Node.js crypto) |
| iOS | Keychain |
| Android | Keystore |
| macOS | Keychain |
| Windows | Credential Vault |

### 7.3 阅后即焚

- **触发**: 打开消息/开始播放
- **时长**: 5s/10s/30s/1m/5m 可选
- **销毁**: 服务端硬删除 + 广播删除事件
- **防重**: `burn_events` 记录防止重复销毁
