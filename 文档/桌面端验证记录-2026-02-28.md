- 验证码登录（发送验证码、错误验证码提示）
- 关键接口与运行日志交叉检查

## 环境信息

- 前端：`apps/desktop`（Vite dev）
- 后端：`apps/backend`（Nest dev）
- 基础设施：PostgreSQL / Redis / MinIO（docker-compose）
- 验证时间：2026-02-28（Asia/Shanghai）

## 功能结论

- 主要链路可用：注册、密码登录、会话创建、消息发送、文件发送、好友申请与同意流程均可完成。
- 存在 3 个重要问题（1 个高优先级安全问题 + 2 个中优先级一致性/稳定性问题），详见下文。

## 问题清单

### 1) [高] 异常日志记录明文验证码（敏感信息泄露）

- 复现步骤：
1. 进入验证码登录，发送验证码。
2. 输入错误验证码触发 `401`。
3. 查看后端日志输出。
- 实际结果：日志会记录请求体 `body`，其中包含 `code`（验证码明文）。
- 影响：验证码属于敏感认证凭据，不应进入错误日志；存在日志泄露风险。
- 相关定位：
  - `apps/backend/src/common/filters/http-exception.filter.ts:40`

### 2) [中] 在线状态 Redis key 类型冲突，导致运行时错误

- 复现步骤：
1. 用户登录（HTTP 登录）。
2. 建立 WebSocket 连接。
3. 查看后端日志。
- 实际结果：出现 `WRONGTYPE Operation against a key holding the wrong kind of value`。
- 原因线索：同一 `online:<userId>` key 在不同模块分别按 `string` 与 `set` 两种数据类型写入。
- 影响：在线状态更新不稳定，可能导致好友在线状态与状态广播异常。
- 相关定位：
  - `apps/backend/src/modules/auth/auth.service.ts:98`（`SET online:<userId> = "1"`）
  - `apps/backend/src/modules/message/gateways/message.gateway.ts:73`（`SADD online:<userId> "online"`）

### 3) [中] 好友关系状态不一致：搜索显示“已是好友”，联系人列表为空

- 复现步骤：
1. 用户 A 与用户 B 有单向 `status=1` 关系记录（历史数据可触发）。
2. 用户 A 在好友页查看列表与搜索 B。
- 实际结果：
  - `GET /friend/list` 返回空；
  - `GET /friend/search` 对同一用户返回 `relation: "friends"`。
- 原因线索：`listFriends` 仅按 `f.user_id = :userId` 查询单向关系，而 `searchUsers` 按双向任一方向映射 `friends`。
- 影响：前端出现“已是好友”但“暂无联系人”的矛盾状态，影响用户判断与操作。
- 相关定位：
  - `apps/backend/src/modules/friend/friend.service.ts:163`
  - `apps/backend/src/modules/friend/friend.service.ts:313`
